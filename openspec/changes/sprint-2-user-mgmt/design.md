## Context

Sprint 2 builds on top of the Sprint 1 initialization by introducing User Management. The application requires administrative dashboards to search, register, modify, and deactivate users, keeping records in the PostgreSQL database. These endpoints and routes must be securely guarded so that only authenticated users with the `ADMINISTRADOR` role can invoke or navigate to them.

## Goals / Non-Goals

**Goals:**
- Implement role-based access validation (middleware `authorizeRole`) on the Express backend.
- Expose REST endpoints under `/api/users` to CRUD users (excluding physical delete).
- Secure the backend endpoints with both `authenticateToken` and `authorizeRole(['ADMINISTRADOR'])`.
- Build the User Management frontend views (table list, search input, pagination controls, and creation/editing modal forms).
- Implement client-side role checks to prevent non-admin users from accessing the user management panel.

**Non-Goals:**
- Implementing equipment or loan models or views.
- Deleting users physically (deactivation changes `activo` flag to `false`).

## Decisions

### 1. Role-Based Access Control Middleware (RBAC)
We will introduce an Express middleware `authorizeRole(allowedRoles: string[])`.
- It will verify the `req.user.rol` property generated by `authenticateToken`.
- If `req.user.rol` is not in the list of allowed roles, the request is rejected with `403 Forbidden`.
*Rationale*: This centralized approach allows us to cleanly reuse the middleware across other routes (such as equipment management) in later sprints.

### 2. REST API Specification for Users
- `POST /api/users` - Create a user. Accepts `usuario`, `password`, `nombre`, `rol`. Returns 201.
- `GET /api/users` - Get paginated list of users. Accepts query parameters `page` (default 1), `limit` (default 10), and `search` (searches `nombre` or `usuario`). Returns 200 with users count and page metadata.
- `PUT /api/users/:id` - Update user details. Accepts `usuario` (optional), `nombre` (optional), `rol` (optional), `activo` (optional). Returns 200.
*Rationale*: A single `PUT /api/users/:id` endpoint can handle editing name/username/role and status toggles (deactivation), keeping the API clean.

### 3. Frontend Layout Integration
We will add a navigation bar or sidebar item to navigate to `/users` on the Dashboard. If a user is not an administrator, the link is hidden, and direct URL access redirects to `/dashboard`.

## Risks / Trade-offs

- **[Risk]** Administrator deactivating themselves.
  - *Mitigation*: The backend will prevent the request if the ID of the user being deactivated matches the ID of the authenticated administrator making the request.
- **[Risk]** Username uniqueness conflicts on edits.
  - *Mitigation*: During edits, the backend check will exclude the current user's ID (`id: { not: userId }`) when checking if the new username already exists.
